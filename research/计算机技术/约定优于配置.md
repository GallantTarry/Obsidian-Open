“约定优于配置”（Convention over Configuration，简称 CoC）是一种现代软件工程和框架设计的核心范式。

它的本质思想非常直接：**系统、框架或工具应该为开发者提供一套“合理的默认规则（约定）”。只要你遵循这套规则，系统就会自动替你打理好一切；只有当你必须打破常规时，才需要去写代码进行“配置”。**

### 🍔 现实生活中的绝妙比喻

为了更直观地理解，我们可以把写代码比作去快餐店点餐：

- **全面配置（没有约定）：** 这就像去吃赛百味（Subway）。你必须事无巨细地告诉店员：要白面包还是全麦、要牛肉还是火鸡、要不要烤一下、生菜放多少、挤什么酱汁……哪怕你只想吃个最普通的汉堡，也得费力地做出一堆选择。
    
- **约定优于配置：** 这就像去吃麦当劳点“巨无霸套餐”。你只需要说“来个套餐”，系统默认（**约定**）给你配好汉堡、中份薯条和中杯可乐。只有当你**不想要默认配置**时，你才会额外开口（即“**配置**”）：把可乐换成黑咖啡。
    

### 💻 编程领域的三大经典案例

在代码世界中，这种理念几乎统治了所有的现代主流框架：

- **项目结构（以 Maven 为例）：**
    
    在早期的构建工具中，你必须写长篇的脚本告诉程序“我的源代码在 A 文件夹，编译后放到 B 文件夹”。而在 Maven 中，有一个隐形**约定**：只要你把代码放在 `src/main/java` 目录下，Maven 就会自动找到并编译它。你不需要写任何一行路径配置。
    
- **Web 框架（以 Spring Boot 为例）：**
    
    传统 Spring 需要写几百行 XML 配置文件。而 Spring Boot 遵循 CoC 理念：当你引入一个 Web 依赖时，它**约定**你一定需要一个 Web 服务器，于是它直接在底层自动帮你启动一个监听 8080 端口的内嵌 Tomcat。只有当你想改用 8081 端口时，才需要去配置文件里写一行 `server.port=8081`。
    
- **现代语言特性（如 Java 21 的无状态 main）：**
    
    正如我们刚才聊到的，现代 JVM 扫描到 `void main()` 时，**约定**这个类需要被无参实例化并启动。它帮你在幕后做完了所有动作，从而消除了传统 `public static` 这种必须显式配置的语法硬要求。
    

### ⚖️ 这种模式的双刃剑效应

|**优势 (Pros)**|**劣势 (Cons)**|
|---|---|
|**极速开发**：消除了大量重复的样板代码，开发者可以把精力全放在业务逻辑上。|**黑盒魔法**：对初学者不友好。如果不知道底层的隐形约定，容易产生“这代码是怎么跑起来的？”的困惑。|
|**易于协作**：全世界的开发者都遵循同一套默认目录和命名规范，接手新项目一目了然。|**排错困难**：一旦不小心违反了某个“隐形约定”（比如文件名拼错了一个字母），系统可能直接罢工，且报错信息往往难以追踪。|

简而言之，“约定优于配置”就是一种“你不说，我就按行业老规矩办”的智能默认机制，是用微小的学习成本（记住约定）来换取巨大的开发效率。
