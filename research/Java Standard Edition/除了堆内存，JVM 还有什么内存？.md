### 一、 除了堆内存，JVM 还有什么内存？

之前我们把堆内存（Heap）比作存放电视机（对象主体）的大仓库。但在 JVM 的世界里，为了让程序井然有序地运行，内存被划分为五大核心区域。除了堆之外，还有以下四个重要区域：

1. **虚拟机栈 (Java Virtual Machine Stack，简称“栈”)**
    
    - **角色：** 专属执行空间（遥控器和临时工）。
        
    - **存什么：** 每当一个方法被调用，栈里就会塞入一个“栈帧”。里面存放的是**局部变量**（比如你定义的 `int i = 1;` 或者对象的引用/遥控器 `A a`）以及方法的调用状态。方法执行完，这块空间立刻释放，速度极快。
        
2. **方法区 (Method Area)**
    
    - **角色：** 图纸与标准车间。
        
    - **存什么：** 这里存放的是整个类的“大蓝图”。比如类名叫什么、有哪些方法、有哪些静态变量（`static` 修饰的）、常量池等。JVM 必须先在方法区读懂类的图纸，才知道在堆里要建一个多大的对象。
        
3. **程序计数器 (Program Counter Register)**
    
    - **角色：** 带路向导。
        
    - **存什么：** 它是内存中最小的一块，专门用来记录**当前线程正在执行哪一条字节码指令**。CPU 切换线程极其频繁，全靠它记住“上一次执行到哪了”，回来才能接着执行。
        
4. **本地方法栈 (Native Method Stack)**
    
    - **角色：** 外援休息室。
        
    - **存什么：** 它的功能和虚拟机栈一模一样，只不过虚拟机栈是给 Java 方法服务的，而本地方法栈是给 **Native 方法**（比如用 C/C++ 实现的底层系统调用）服务的。
        

### 二、 谜底揭晓：`<init>` 究竟是个什么东西？

少侠在写 Java 代码时，确实**从来不会**写下 `<init>` 这个名字。你没见过它非常正常，因为**它是 Java 编译器在幕后偷偷生成的“暗号”**。

在字节码（`.class` 文件）的视角里，根本没有所谓的“构造器（Constructor）”，取而代之的就是 **`<init>` 方法**（专业叫法是：**实例初始化方法**）。

#### 为什么需要 `<init>`？

你在写一个类的时候，初始化成员变量的方法有很多种，比如：

1. 在声明变量时直接赋值：`int age = 18;`
    
2. 写在实例代码块里：`{ age = 20; }`
    
3. 写在构造器里：`public Person() { age = 22; }`
    

这三种赋值方式如果同时存在，JVM 到底该听谁的？执行顺序又该如何？

为了不打架，Java 编译器在编译时，会把这三处的代码**打包、融合成一个标准的方法**，这个方法在字节码里的名字就叫 **`<init>`**。

#### `<init>` 内部的执行大总管：

当一辆“毛坯车”在堆里建好后，`<init>` 方法就会被调用，它内部严格按照以下顺序执行：

1. **先调用父类的 `<init>` 方法**（也就是 `super()`，确保父类先初始化完）。
    
2. **执行自己类中变量的直接赋值语句和实例代码块**（按它们在代码里出现的先后顺序）。
    
3. **最后执行你自己在构造器大括号里写的代码**。
    

所以，`<init>` 就是**构造器的完全体和真面目**，它把零散的初始化代码整合到了一起。
